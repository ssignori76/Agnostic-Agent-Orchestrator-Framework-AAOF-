# Guide: Requirements Checklist and Contract Check

## Overview

The **Requirements Checklist** is a core AAOF mechanism that prevents silent omissions
during implementation. It ensures that every feature and module declared in the
execution plan at STEP 2 has been **concretely implemented** before the workflow
advances to STEP 5 (Validation & Test).

The **Contract Check** is the automated verification run at the end of STEP 4 that
reads the requirements checklist and confirms every requirement has a corresponding
artifact (file, module, or endpoint) in `output/`.

---

## How It Works

```
STEP 2 (Execution Plan)
  └─ User says "GO"
  └─ Agent generates session/requirements_checklist.json
        (all requirements with status: "pending")

STEP 4 (Implementation)
  └─ Agent writes code in output/
  └─ At the end of STEP 4:
       ├─ Agent runs Contract Check
       ├─ Each requirement → verifies artifacts exist
       ├─ Updates status: "pass" or "fail"
       └─ Sets VAR_CONTRACT_CHECK: "PASS" or "FAIL"

If VAR_CONTRACT_CHECK == "FAIL":
  └─ ⛔ BLOCKED — agent reports missing requirements
  └─ Agent does NOT advance to STEP 5

If VAR_CONTRACT_CHECK == "PASS":
  └─ ✅ Gate POST-4.4 satisfied → proceed to STEP 5
```

---

## The `session/requirements_checklist.json` File

This file is generated by the agent automatically. You do not need to create it manually.
However, understanding its structure helps you guide the agent when the execution plan
is complex.

### Structure

```json
{
  "checklist_version": "1.0",
  "generated_at": "<ISO-8601 timestamp>",
  "step": 2,
  "execution_plan_summary": "Brief summary of the agreed execution plan",
  "requirements": [
    {
      "id": "REQ-001",
      "step": 4,
      "category": "functional",
      "description": "What this requirement is",
      "artifacts": ["output/src/module/file.js"],
      "verification": "How to confirm it is implemented",
      "status": "pending",
      "evidence": ""
    }
  ]
}
```

### Requirement Categories

| Category | What it covers |
|----------|----------------|
| `functional` | Business logic features (endpoints, modules, user flows) |
| `infrastructure` | Docker, compose files, database migrations, CI/CD |
| `security` | Auth middleware, input validation, CORS, secrets handling |
| `testing` | Test files and coverage obligations |

## Guiding the Agent for Better Checklists

When providing the specification in `specs/active/`, be explicit about:

1. **Feature modules** — list every distinct feature by name
2. **Expected file structure** — name the files/modules you expect
3. **Acceptance criteria** — define what "done" means for each feature

### Example Specification (Vulnerable Lab)

```markdown
## Features to implement

### 1. Authentication
- POST /api/auth/register — hash password with bcrypt, return JWT
- POST /api/auth/login — validate credentials, return JWT
- Files: src/auth/register.js, src/auth/login.js, src/auth/authRoutes.js

### 2. Product Listing
- GET /api/products — paginated, public
- Files: src/products/productController.js, src/products/productRoutes.js

### 3. Shopping Cart (REQUIRED — do not omit)
- POST /api/cart/add — add item, requires auth
- GET /api/cart — list items for current user
- DELETE /api/cart/:itemId — remove item
- Files: src/cart/cartController.js, src/cart/cartRoutes.js
```

With this level of detail, the agent will generate a checklist that accurately
maps each feature to the expected files, making it impossible to "forget" the
shopping cart during implementation.

## What Happens When the Contract Check Fails

If the agent finds unimplemented requirements at the end of STEP 4, it will stop
and display a report like this:

```
⛔ Contract Check FAILED — transition STEP 4 → STEP 5 blocked.

The following requirements declared in the execution plan have not been implemented:

  ❌ REQ-004: Shopping cart module (add/list/remove)
     Missing artifacts:
       - output/src/cart/cartController.js (not found)
       - output/src/cart/cartRoutes.js (not found)
     Verification criterion: Both files exist and cartRoutes.js has POST /add,
     GET /, DELETE /:itemId routes with auth middleware.

Please implement the missing requirements before advancing to STEP 5.
```

You can then:
- **Ask the agent to implement the missing feature** and re-run the check
- **Waive a requirement** if it was intentionally deferred (the agent will
  mark it as `"waived"` and allow progression)

---

## Waiving a Requirement

If you intentionally want to skip a requirement for this iteration, tell the agent:

> "Waive REQ-004 — the shopping cart will be implemented in the next sprint."

The agent will:
1. Set `status` to `"waived"` for REQ-004
2. Record your reason in the `evidence` field
3. Exclude it from the failure count
4. Re-check — if no other failures remain, set `VAR_CONTRACT_CHECK` to `"PASS"`

**Important:** Waivers are recorded in `session/requirements_checklist.json` and
logged in `session/step_evidence.json` for full traceability.

---

## Reviewing the Contract Check Results

After STEP 4 completes, you can inspect:

1. **`session/requirements_checklist.json`** — full per-requirement status with evidence
2. **`session/step_evidence.json`** — the `contract_check` block in the STEP 4→5 transition:

```json
{
  "from": 4,
  "to": 5,
  "contract_check": {
    "total_requirements": 10,
    "passed": 9,
    "failed": 0,
    "waived": 1,
    "result": "PASS",
    "failed_ids": [],
    "waived_ids": ["REQ-004"]
  }
}
```

---

## Contract Check on Retry (STEP 6 → STEP 4)

When the agent retries STEP 4 after a STEP 6 rollback:

- The checklist is **not reset** entirely — requirements already `"pass"` are preserved
- Only `"fail"` requirements are reset to `"pending"` for re-verification
- This means incremental progress is tracked across retries

## Frequently Asked Questions

**Q: What if the execution plan doesn't specify exact file names?**

The agent infers reasonable file paths based on declared module names and the
project's naming conventions. The inferred paths are documented in the `verification`
field of each requirement. You can review them after the checklist is generated.

**Q: Can I add requirements manually?**

Yes. You can edit `session/requirements_checklist.json` between STEP 2 and STEP 4
to add, modify, or remove requirements. The agent will respect the file as-is when
running the contract check. Inform the agent after editing so it re-reads the file.

**Q: What if a requirement maps to multiple files, but only some are missing?**

The contract check fails the requirement if ANY of its declared artifacts is missing.
The `evidence` field will specify exactly which files were found and which were not.

**Q: Does the contract check replace testing?**

No. The contract check only verifies that implementation **artifacts exist**. It does
not test runtime behavior, correctness, or edge cases. That remains the responsibility
of STEP 5 (Validation & Test). Think of the contract check as a completeness gate —
"did you build everything you said you would?" — not a correctness gate.
