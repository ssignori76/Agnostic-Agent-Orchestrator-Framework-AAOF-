# Development Rules

> **Purpose:** Code quality standards, file organization, naming conventions, and
> modular architecture principles for all code generated by the Agnostic Orchestrator
> Agent.
>
> **Scope:** Language-agnostic. These rules apply to all output regardless of stack.

---

## 1. File Size Limits

Prefer small, focused files. Apply the following differentiated limits:

| File Type | Limit | Principle |
| :--- | :--- | :--- |
| Source code (classes, modules) | **max 200 lines** | One class / one responsibility |
| Dockerfiles | **no rigid limit** | Use multi-stage; clarity over brevity |
| docker-compose.yml | **no rigid limit** | Prefer 1 compose file per environment |
| Kubernetes manifests | **1 resource per file** | Logical separation, not line counting |
| Config files (JSON, YAML, TOML) | **no rigid limit** | Data, not logic |
| Markdown documentation | **max 300 lines** | Split by topic if longer |

> **Principle:** Prefer small, focused files. For source code, max 200 lines per file.
> For infrastructure and configuration, prefer logical separation (1 resource per file)
> over line counting.

---

## 2. File Header Requirements

Every source code file must begin with a header block:

```
# File: <filename>
# Purpose: <one-line description of what this file does>
# Dependencies: <list of files, modules, or services this file depends on>
# Created: <YYYY-MM-DD>
# Modified: <YYYY-MM-DD>
```

For compiled languages (Java, Go, C#), use the language-native comment style.

---

## 3. Naming Conventions

| Entity | Convention | Example |
| :--- | :--- | :--- |
| Files (source) | `snake_case` | `user_service.py` |
| Files (config) | `kebab-case` | `docker-compose.dev.yml` |
| Directories | `lowercase` | `src/`, `handlers/` |
| Classes | `PascalCase` | `UserService` |
| Functions / methods | `camelCase` (JS/TS) or `snake_case` (Python) | `getUserById` |
| Constants | `UPPER_SNAKE_CASE` | `MAX_RETRY_COUNT` |
| Environment variables | `UPPER_SNAKE_CASE` | `DATABASE_URL` |
| Docker images | `lowercase-hyphen` | `my-app-api:1.0.0` |

---

## 4. Modularity Principles

- **Single Responsibility:** Each file or module does exactly one thing.
- **No God Files:** If a file is doing too many things, split it.
- **Dependency Direction:** Higher-level modules must not depend on lower-level
  implementation details. Use interfaces/contracts.
- **Entry Points:** Each service must have a clear, documented entry point
  (e.g., `main.py`, `index.js`, `Main.java`).
- **Configuration Externalization:** No hardcoded values in source code. Use
  environment variables or config files.

---

## 5. Comment and Documentation Standards

- Write comments to explain **why**, not **what**. The code explains what.
- Every public function/method must have a docstring or JSDoc-style comment.
- Complex algorithms must have an inline explanation.
- TODO comments must include a reference: `# TODO(AAOF-123): fix edge case`
- Avoid commented-out code in committed files. Use git history instead.

---

## 6. Code Quality Standards

- **No Dead Code:** Remove unused imports, variables, and functions.
- **Error Handling:** Every operation that can fail must handle the error explicitly.
  See `error_handling_rules.md`.
- **Logging:** Use structured logging (JSON format preferred). Log at appropriate
  levels: DEBUG, INFO, WARN, ERROR.
- **Avoid Magic Numbers:** Define constants with meaningful names.
- **Idempotency:** Operations should be safely re-runnable where possible.

---

## 7. Language-Agnostic Principles

These apply regardless of the programming language used:

1. **Fail fast:** Validate inputs at the earliest possible point.
2. **Immutability preferred:** Prefer immutable data structures where practical.
3. **Separation of concerns:** UI, business logic, and data access must be separate.
4. **12-Factor App compliance:** Follow [12factor.net](https://12factor.net) principles
   for all services.
5. **Graceful shutdown:** Services must handle SIGTERM and shut down cleanly.
6. **Health endpoints:** Every HTTP service must expose `/health` or `/healthz`.
